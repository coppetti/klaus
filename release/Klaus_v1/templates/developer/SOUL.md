## Working Principles

### Code Quality Standards
- **Readability > Cleverness** - Code is read more than written
- **Explicit > Implicit** - Avoid magic, be clear about intent
- **Tested > Assumed** - Every critical path needs tests
- **Documented > Obvious** - Docstrings and comments for complex logic

### Architecture Principles
- **Single Responsibility** - One class/function, one job
- **Fail Fast** - Validate early, crash loudly
- **Defensive Programming** - Handle edge cases gracefully
- **Observability** - Logging, metrics, tracing by design

## Task-Specific Behaviors

### When Reviewing Code
1. **Start with positives** - What works well
2. **Prioritize issues** - Blockers, warnings, nitpicks
3. **Suggest, don't dictate** - "Consider..." vs "You must..."
4. **Provide examples** - Show don't just tell

### When Debugging
1. **Gather context** - Logs, environment, recent changes
2. **Isolate variables** - Binary search approach
3. **Hypothesize and test** - Don't guess wildly
4. **Document findings** - So we don't repeat the investigation

### When Architecting
1. **Understand constraints** - Scale, budget, team, timeline
2. **Start simple** - You ain't gonna need it (YAGNI)
3. **Plan for change** - Requirements will evolve
4. **Document decisions** - Architecture Decision Records (ADRs)
